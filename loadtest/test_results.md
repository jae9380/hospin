> 병목 레이어 O 판정 기준: Application 평균 > 50ms (경고 임계치 초과)

#### 도메인별 회차별 병목 분석

- **hospital — O / O / O**
  - Run 1: Application 176ms, Infrastructure 176ms → 두 레이어가 동일하게 높음. Presentation p95 1810ms는 JVM Cold Start + Security Filter 첫 실행 오버헤드 의심
  - Run 2: Application 53.4ms, Infrastructure 53.3ms → 임계치(50ms) 소폭 초과. Application ≈ Infrastructure로 Haversine 연산과 DB 쿼리 비용이 혼재하는 케이스 2 경계 의심
  - Run 3: Run 2와 동일 패턴 (54.6ms / 54.5ms) → 재현성 확인, 구조적 문제로 판단

- **schedule — O / X / X**
  - Run 1: Application 148ms (임계치 3배) — Presentation p95 1490ms는 Cold Start 의심, Application 자체도 높아 스케줄 충돌 검증 로직 또는 날짜 계산 처리 과다 의심
  - Run 2: Application 29.2ms, Infrastructure 4.46ms → 정상 범위. App/Infra 비율 6.5x로 로직 비중이 높긴 하나 절대값은 임계치 이하
  - Run 3: Run 2와 동일 패턴 (31.0ms / 6.65ms) → Run 1은 Cold Start 이상값, 실제 병목은 경미한 수준으로 판단

- **member — O / X / X**
  - Run 1: Application 106ms (임계치 2배 이상) — Infrastructure 15.3ms로 DB는 문제 없음. BCrypt 해시 연산 비용이 Application 시간을 지배하는 것으로 의심. Presentation p95 1990ms는 Cold Start 의심
  - Run 2: Application 39.7ms, Infrastructure 0.902ms → 절대값은 임계치 이하이나 App/Infra 비율 44x → BCrypt 연산이 여전히 지배적, 부하 증가 시 임계치 초과 가능성 있음
  - Run 3: Run 2와 동일 패턴 (39.5ms / 0.913ms) → 재현성 확인

> Grafana 패널 매핑 (Layer 섹션 → 도메인별 stat 패널)
>
> - hospital: [Hospital] Presentation p95 (ms) / [Hospital] Application 평균 (ms) / [Hospital] Infrastructure 평균 (ms)
> - schedule: [Schedule] Presentation p95 (ms) / [Schedule] Application 평균 (ms) / [Schedule] Infrastructure 평균 (ms)
> - member: [Member] Presentation p95 (ms) / [Member] Application 평균 (ms) / [Member] Infrastructure 평균 (ms)

> 병목 판단 기준
>
> - Infrastructure > 100ms → DB 쿼리 병목 (인덱스 추가 또는 쿼리 최적화)
> - Application > 50ms → 비즈니스 로직 병목 (알고리즘 개선, /nearby Haversine 계산 예상)
> - Presentation > Application + Infrastructure 합 → AOP / JSON 직렬화 / Security Filter 체인

### Layer 섹션 (Hexagonal 레이어 분석)

| 패널명                                    |  정상   |     경고      |  위험   | 기록 항목            |
| ----------------------------------------- | :-----: | :-----------: | :-----: | -------------------- |
| Presentation p95 응답 시간                | < 500ms |  500ms ~ 1s   |  > 1s   | 엔드포인트별         |
| Application 평균 비즈니스 처리 시간       | < 50ms  | 50ms ~ 200ms  | > 200ms | 레이어 처리 오버헤드 |
| Infrastructure 외부 의존성 평균 응답 시간 | < 100ms | 100ms ~ 300ms | > 300ms | DB/외부 의존성 성능  |
| Application 처리 지연 (Presentation 대비) | < 50ms  | 50ms ~ 200ms  | > 200ms | AOP 오버헤드 참고    |
| Presentation 실패율                       |  < 1%   |    1% ~ 5%    |  > 5%   | 시나리오별           |

> 병목 레이어 해석
>
> ```
> [케이스 1] Infrastructure 평균 >> Application 평균
>   → DB 쿼리 병목. 인덱스 추가 또는 쿼리 최적화 필요
>
> [케이스 2] Application 평균 >> Infrastructure 평균
>   → 비즈니스 로직 병목. 알고리즘 개선 필요
>   → GET /api/hospital/nearby 에서 이 패턴 예상 (Haversine 전체 계산)
>
> [케이스 3] Presentation p95 >> (Application + Infrastructure) 합
>   → AOP 오버헤드, JSON 직렬화, Security Filter 체인 확인
> ```
