## 2차 리팩토링

### findByUserId에서의 비원자적 흐름

`findByUserId 조회 → ifPresentOrElse 분기 → save`
해당 흐름에서 `RefreshToken` 엔티티의 `userId` 컬럼에 `unique = true` 제약이 존재한다. 여기서 두 쓰레드가 동일 데이터로 로그인 할 경우 문제가 발생된다.

```
Thread A                               Thread B
findByUserId(1L) → Optional.empty
                                       findByUserId(1L) → Optional.empty
save(RefreshToken(1L, tokenA)) ✅
                                       save(RefreshToken(1L, tokenB))
                                       → DataIntegrityViolationException 💥
```

해당 문제를 해결하기 위해 `upsert`을 작성

### @Monitored 오표기

Controller 레이어에 존재하는 메서드 선언부에 `(layer = "application")`로 오표기 되어 있다.

controller에서 실패가 발생하여도 다른 레이어에 실패로 로그가 남는다.

### Login 로직 내 중복 조회

로그인 시 하나의 요청에서 `AuthenticationServiceImpl`, `MemberDomainServiceImpl`에서 각각 동일 데이터를 2번 조회를 한다는 것을 확인하였다.

이는 HikariCP 커넥션을 불필요하게 1회 추가적으로 낭비하게 된다. 그래서 기존 70U와 100U에서 이미 커넥션이 부족한 상황에서 문제를 더 악화 시킨다는 것을 판단할 수 있다.

### BCrypt strength

기존 코드
**`EncoderConfig.java`**

```java
@Bean
public BCryptPasswordEncoder bCryptPasswordEncoder() {
    return new BCryptPasswordEncoder();
}
```

여기서 strength에 대해서 아무 설정 없이 기본 설정값인 10을 사용하고 있다. 이는 strength 10에서 단일 해시 연산에 약 60~100ms의 CPU 시간이 소요된다는 내용을 찾았다.

그래서 70~100U에서 동시에 로그인하면 Tomcat 스레드가 BCrypt 연산으로 동시에 점유되어 다른 요청의 스레드 대기가 발생할 것을 의심했다.

strength 기본(10) -> 8으로 조정
